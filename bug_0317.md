# Auto-Formatting Indent Bugs - 2026-03-17

> Ultimate Boss Script (`test/oracle_format_ultimate_boss.sql`) 포맷팅 결과에서
> 문맥/괄호 상태 대비 indent가 예상과 다른 부분을 전수 검토한 결과

---

## BUG 5 (심각): `EXTRACT(YEAR FROM ...)` - `FROM`이 괄호 안에서 절 키워드로 탈출

`EXTRACT(YEAR FROM ...)` 내부의 `FROM`이 SQL `FROM` 절로 인식되어 indent level 0으로 점프.
`suppress_comma_break_depth > 0` (괄호 안)임에도 `FROM`은 clause keyword로서 줄바꿈이 억제되지 않음.

```sql
-- 원본 (INSERT INTO qt_fmt_cal SELECT ...)
    EXTRACT(YEAR FROM d.dt)  AS yyyy,
    EXTRACT(MONTH FROM d.dt) AS mm,
    EXTRACT(DAY FROM d.dt)   AS dd

-- 포맷팅 결과: FROM이 indent 0으로 탈출
    EXTRACT (YEAR
FROM d.dt) AS yyyy,
    EXTRACT (MONTH
FROM d.dt) AS mm,
    EXTRACT (DAY
FROM d.dt) AS dd
```

- **발생 위치**: INSERT INTO qt_fmt_cal (L205), QUERY 7 MODEL 소스 서브쿼리 (L558-569)
- **원인**: `FROM`이 `FORMAT_CLAUSE_KEYWORDS`에 있고, `EXTRACT(` 내부 context를 구분하지 않음. `suppress_comma_break_depth > 0` 상태에서 comma break만 억제되고, clause keyword break는 억제되지 않음
- **영향**: **SELECT 리스트 indent 전체 붕괴** - 후속 컬럼들의 indent가 모두 틀어짐

## BUG 6: `RETURNING` inside JSON 함수 - 괄호 안에서 절 키워드로 탈출

`JSON_VALUE(... RETURNING ...)`, `JSON_OBJECT(... RETURNING CLOB)`, `JSON_ARRAYAGG(... RETURNING CLOB)` 내부의
`RETURNING`이 SQL `RETURNING` 절로 인식되어 괄호 밖 indent level로 줄바꿈.

```sql
-- 원본 (VIEW 내 CTE)
        JSON_VALUE(e.json_profile, '$.level' RETURNING VARCHAR2(30)) AS profile_level

-- 포맷팅 결과: RETURNING이 CTE indent level 0으로 탈출
        JSON_VALUE (e.json_profile, '$.level'
    RETURNING VARCHAR2 (30)) AS profile_level
```

```sql
-- 원본 (VIEW SELECT의 JSON_OBJECT)
    JSON_OBJECT('empId' VALUE b.emp_id, ... RETURNING CLOB) AS profile_json

-- 포맷팅 결과: RETURNING이 top-level FROM 절 indent로 탈출
    )
RETURNING CLOB) AS profile_json
```

```sql
-- 원본 (QUERY 8 JSON_ARRAYAGG)
    JSON_ARRAYAGG(JSON_OBJECT(... RETURNING CLOB) ORDER BY ... RETURNING CLOB) AS dept_emp_json

-- 포맷팅 결과: 두 RETURNING 모두 탈출
    ...
RETURNING CLOB) ORDER BY e.salary DESC, e.emp_id
RETURNING CLOB) AS dept_emp_json
```

```sql
-- 원본 (FINAL SANITY QUERY)
    JSON_VALUE(e.json_profile, '$.level' RETURNING VARCHAR2(30))

-- 포맷팅 결과
    JSON_VALUE (e.json_profile, '$.level'
RETURNING VARCHAR2 (30))
```

- **발생 위치**: VIEW (L253-256, L307), QUERY 8 (L592-593), PACKAGE BODY (L730, L736), FINAL QUERY (L982-983)
- **원인**: BUG 5와 동일 메커니즘. `RETURNING`이 `FORMAT_CLAUSE_KEYWORDS`에 있고, 괄호 안 여부를 clause keyword 분기에서 체크하지 않음
- **영향**: JSON 함수 사용하는 모든 쿼리에서 indent 파괴 + `)` 위치와 시각적 불일치

## BUG 7: `CROSS APPLY`가 이전 `ON` 조건에 붙음

`JOIN ... ON` 뒤에 오는 `CROSS APPLY`가 별도 줄로 분리되지 않고 `ON` 조건 끝에 연결됨.

```sql
-- 원본 (QUERY 1)
FROM org_enriched oe
JOIN qt_fmt_dept d
    ON d.dept_id = oe.dept_id
CROSS APPLY (
    SELECT ...
) ca

-- 포맷팅 결과: CROSS APPLY가 ON 줄에 붙음
JOIN qt_fmt_dept d
    ON d.dept_id = oe.dept_id CROSS APPLY (
        SELECT ...
    ) ca
```

- **발생 위치**: QUERY 1 (L418)
- **원인**: `CROSS`가 join modifier로 처리되지만, 앞에 `ON` 조건이 끝난 상태에서 새로운 절로 인식 안 됨
- **영향**: JOIN 구조 가독성 저하

## BUG 8: `SET` in SEARCH/CYCLE context가 SQL `SET` 절로 오인

`SEARCH DEPTH FIRST BY col SET ordering` / `CYCLE col SET flag TO ...` 에서
`SET`이 SQL `SET` 절(UPDATE SET)로 인식되어 불필요한 줄바꿈 발생.

```sql
-- 원본 (QUERY 3)
SEARCH DEPTH FIRST BY dept_name SET dfs_order
CYCLE dept_id SET is_cycle TO 'Y' DEFAULT 'N'

-- 포맷팅 결과: SET이 절 키워드로 분리
) SEARCH DEPTH FIRST BY dept_name
SET dfs_order CYCLE dept_id
SET is_cycle TO 'Y' DEFAULT 'N'
```

- **발생 위치**: QUERY 3 (L483-485)
- **원인**: `SET`이 `FORMAT_CLAUSE_KEYWORDS`에 포함, SEARCH/CYCLE 문맥 구분 없음
- **영향**: recursive CTE의 SEARCH/CYCLE 구문 indent 파괴

## BUG 9: MATCH_RECOGNIZE 내부 하위절 미분리

MATCH_RECOGNIZE `(...)` 블록 안의 `MEASURES`, `PATTERN`, `DEFINE`, `ALL ROWS PER MATCH` 등이
별도 줄로 분리되지 않고 이전 내용에 연결됨.

```sql
-- 원본
MATCH_RECOGNIZE (
    PARTITION BY emp_id
    ORDER BY sale_date, sale_id
    MEASURES
        MATCH_NUMBER() AS match_no, ...
    ALL ROWS PER MATCH
    PATTERN (low+ mid* high)
    DEFINE
        low  AS (...), ...
)

-- 포맷팅 결과: ORDER BY 이후 모두 한 줄로 연결
MATCH_RECOGNIZE (
    PARTITION BY emp_id
    ORDER BY sale_date,
        sale_id MEASURES MATCH_NUMBER () AS match_no,
        ...ALL ROWS PER MATCH PATTERN (low + mid * high) DEFINE low AS ...,
        mid AS ...,
        high AS ...
)
```

- **발생 위치**: QUERY 6 (L535-542)
- **원인**: MATCH_RECOGNIZE 내부를 subquery-like block으로 처리하지만 `MEASURES`, `PATTERN`, `DEFINE` 등이 clause keyword로 등록되어 있지 않아 줄바꿈 트리거 없음
- **영향**: MATCH_RECOGNIZE 블록 내부 구조 완전 소실

## BUG 10: MODEL 하위절 미분리

`MODEL` 절 이후 `PARTITION BY`, `DIMENSION BY`, `MEASURES`, `RULES` 등이 별도 줄로 분리되지 않음.

```sql
-- 원본
MODEL
    PARTITION BY (year_key, channel_code)
    DIMENSION BY (month_key)
    MEASURES (base_amt, proj_amt)
    RULES SEQUENTIAL ORDER (...)

-- 포맷팅 결과: 전부 한 줄
MODEL PARTITION BY (...) DIMENSION BY (...) MEASURES (...) RULES SEQUENTIAL ORDER (...)
```

- **발생 위치**: QUERY 7 (L572)
- **원인**: MODEL 고유 하위 키워드가 clause keyword로 등록되어 있지 않음
- **영향**: MODEL 절 내부 구조 소실

## BUG 11: PIVOT / UNPIVOT `(...)` 내용 한 줄 collapse

PIVOT, UNPIVOT 뒤의 `(...)` 내용이 모두 한 줄로 압축됨.

```sql
-- 원본 (QUERY 4)
PIVOT (
    SUM(net_amt)
    FOR channel_code IN (
        'ONLINE'  AS online_amt,
        'DIRECT'  AS direct_amt,
        'PARTNER' AS partner_amt
    )
)

-- 포맷팅 결과
) PIVOT (SUM (net_amt) FOR channel_code IN ('ONLINE' AS online_amt, 'DIRECT' AS direct_amt, 'PARTNER' AS partner_amt))
```

```sql
-- 원본 (QUERY 5)
UNPIVOT INCLUDE NULLS (
    comp_value FOR comp_type IN (
        salary     AS 'SALARY',
        bonus      AS 'BONUS',
        total_comp AS 'TOTAL_COMP'
    )
)

-- 포맷팅 결과
) UNPIVOT INCLUDE NULLS (comp_value FOR comp_type IN (salary AS 'SALARY', bonus AS 'BONUS', total_comp AS 'TOTAL_COMP'))
```

- **발생 위치**: QUERY 4 (L507), QUERY 5 (L523)
- **원인**: PIVOT/UNPIVOT 뒤의 `(`가 subquery로 인식되지 않아 `suppress_comma_break_depth` 증가 → 내부 comma/keyword 모두 줄바꿈 억제
- **영향**: PIVOT/UNPIVOT 블록 가독성 소실

## BUG 12: PL/SQL `FOR r IN (WITH ... SELECT ...)` 내부 indent 혼란

패키지 내 `FOR r IN (cursor query)` 에서 WITH + SELECT가 포함된 쿼리의 indent가
일관되지 않음. SELECT 항목과 FROM/WHERE/GROUP BY 절의 depth가 뒤섞임.

```sql
-- 원본 (PACKAGE BODY print_dept_rollup)
        FOR r IN (
            WITH dept_tree (...) AS (
                SELECT
                    d.dept_id,
                    d.parent_dept_id,
                ...
            )
            SELECT
                dt.dept_id,
                dt.dept_name,
            ...
            FROM dept_tree dt
            LEFT JOIN qt_fmt_emp e
                ON e.dept_id = dt.dept_id
            ...
        ) LOOP

-- 포맷팅 결과: indent depth가 뒤섞임
        FOR r IN (
                WITH dept_tree (...) AS (
                        SELECT
                    d.dept_id,           -- ← SELECT item이 SELECT보다 덜 들여쓰기됨
                    d.parent_dept_id,
                ...
                        FROM qt_fmt_dept d
                        WHERE d.dept_id = p_root_dept_id
                        UNION ALL
                        SELECT
                    c.dept_id,           -- ← 동일한 문제
                ...
                )
                SELECT
                dt.dept_id,              -- ← 동일한 문제
                dt.dept_name,
                ...
                FROM dept_tree dt
            LEFT JOIN qt_fmt_emp e       -- ← LEFT JOIN이 FROM보다 덜 들여쓰기됨
                ON e.dept_id = dt.dept_id
                AND e.salary >= p_min_salary
                GROUP BY dt.dept_id,
                dt.dept_name,            -- ← GROUP BY items이 제대로 정렬 안 됨
        ) LOOP
```

- **발생 위치**: PACKAGE BODY print_dept_rollup (L750-787)
- **원인**: PL/SQL `FOR ... IN (cursor_subquery)` 내부의 WITH/SELECT가 중첩된 paren context에서 indent level 계산이 올바르지 않음. subquery indent와 CTE indent가 겹침
- **영향**: PL/SQL 내 cursor FOR loop의 복잡한 쿼리 가독성 심각하게 저하

## BUG 13: PL/SQL CASE WHEN 값이 WHEN과 동일 indent

PL/SQL 블록 내 `EXECUTE IMMEDIATE CASE` 및 `ORDER BY CASE`에서 THEN 뒤의 값이
WHEN 키워드와 동일한 indent level에 위치.

```sql
-- 원본 (CLEANUP block ORDER BY CASE)
        ORDER BY
            CASE object_type
                WHEN 'VIEW' THEN 1
                WHEN 'PACKAGE BODY' THEN 2
                ...
            END

-- 포맷팅 결과: THEN 값이 다음 줄에 WHEN과 같은 depth로
        ORDER BY
            CASE object_type
                WHEN 'VIEW' THEN
                1                     -- ← WHEN과 같은 indent level
                WHEN 'PACKAGE BODY' THEN
                2
                ...
            END
```

```sql
-- 원본 (EXECUTE IMMEDIATE CASE)
            EXECUTE IMMEDIATE
                CASE
                    WHEN r.object_type = 'VIEW' THEN
                        'DROP VIEW ' || r.object_name
                    ...
                END;

-- 포맷팅 결과: THEN 값이 WHEN과 동일 indent
            EXECUTE IMMEDIATE
            CASE
                    WHEN r.object_type = 'VIEW' THEN
                    'DROP VIEW ' || r.object_name      -- ← WHEN과 같은 indent
                    ...
            END;
```

- **발생 위치**: CLEANUP block (L22-47)
- **원인**: PL/SQL CASE의 THEN 뒤 값에 대한 추가 indent 처리 없음
- **영향**: CASE 분기 가독성 저하 (WHEN 조건과 결과값의 시각적 구분 불가)

## 공통 패턴

BUG 5, 6, 8은 모두 **같은 근본 원인**을 공유:

> **괄호(`()`) 안에서 clause keyword가 줄바꿈을 발생시키는 것을 억제하지 않음**

현재 `suppress_comma_break_depth > 0` 상태에서:
- comma break → 억제됨 (정상)
- clause keyword break → **억제 안 됨** (버그)

`FROM`, `RETURNING`, `SET` 등이 `FORMAT_CLAUSE_KEYWORDS`에 포함되어 있어서,
함수 호출 `func(... FROM ...)` 이나 `func(... RETURNING ...)` 내부에서도 절 분리가 발생함.
